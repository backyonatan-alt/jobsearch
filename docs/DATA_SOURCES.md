# Data sources — how Pursuit derives what it shows

Every number/insight in the UI comes from the user's own applications (+ the
AI interviewer brief). Nothing is fabricated. This pins where each surface gets
its data so the next change of that shape doesn't drift.

## Status model
`wishlist · applied · screen · interview · offer · rejected · withdrawn`.
- **Active / "in progress"** = `applied, screen, interview, offer`.
- **Stale / "gone quiet"** = status in `applied, screen` AND days since last
  activity ≥ 7, where *last activity* = `max(applied_at, last_follow_up_at)`
  (see `isStale` in `web/src/lib/app-helpers.js`). Logging a follow-up bumps
  `last_follow_up_at`, which resets the clock. Display of "Applied X ago" still
  uses `applied_at`.

## Today

### Stat strip (under the greeting)
- **In progress** = count of active apps (above).
- **Awaiting reply** = `applied`/`screen` with no upcoming interview event.
- **Gone quiet** = `isStale` apps.

### Brief (when there's an upcoming interview)
`upcomingEvents` = for apps in `screen/interview/offer`, the future interview
events (from `/interviews`) merged with the dossier's linked `meeting`,
de-duped, soonest first. `nextInterview` = the first one.
- **Lede / meta** = that event (company, role, interviewer, duration, medium, panel).
- **Baby-blue insight** = `dossier.content.snapshot` (→ `style.lead` → `background`).
- **Worth reviewing** = `dossier.content.lands[0:3]`.
- **Two quick tips** = `dossier.content.company.watch_fors[0:2]` (fallback: extra `lands`).
- **Later this week** = the remaining `upcomingEvents`.
If there's no upcoming interview or no brief, this whole block is hidden.

### No-interview state ("What you can do today" + "Recently added")
Suggestions are derived from real signals, priority order, capped at 4 — only
shown when they apply:
1. `screen`/`interview` app with no dossier → **Generate prep**.
2. `isStale` app → **Log a follow-up**.
3. `offer` app → **Decide on the offer**.
4. `screen`/`interview` app missing `hiring_manager_name` → **Add hiring manager**.

**Recently added** = apps with a `created_at`, sorted desc, top 5.

## Application detail / Interview prep
The interview prep (dossier) is generated by Claude with web search
(`POST /api/applications/{id}/dossier/refresh`) and stored. The page maps:
`dossier.interviewer_name`, `dossier.generatedAgo`, `dossier.meeting.*`, and
`dossier.content.*` (`interviewer`, `snapshot`, `style{lead,tells}`, `lands`,
`avoid`, `signals`, `questions`, `company{blurb,direction,facts,process,watch_fors}`).
Sections with no data are omitted. The company brief, AI tips box, and next-
interview card all read from these fields.

## Follow-ups
`POST /api/applications/{id}/follow-ups` records `{note, channel, occurred_at}`
into the `follow_ups` table (migration 0013). It appears in the activity
timeline and resets the quiet clock (`last_follow_up_at` = max occurred_at,
surfaced on the application payload). **It never sends a message** — Pursuit
advises, it does not act on the user's behalf.

## Insights (`/app/funnel`)
- **Pipeline funnel** (cumulative reach): `applied` = all non-wishlist;
  `screen` = screen+interview+offer; `interview` = interview+offer; `offer` = offer.
  The **conversion %** next to each = stage count ÷ applied (top of funnel = 100%).
- **Reply rate** = apps that reached `screen` or further ÷ all non-wishlist.
- **Avg. time to first reply** = ⚠️ **proxy**: mean of `daysSince(applied_at)` for
  apps currently at `screen/interview/offer`. We don't store reply timestamps
  yet, so this is an estimate and is labeled as such in the UI.
- **Furthest stage reached** = walk `offer → interview → screen → applied`, first match.
- **Application activity** = 12 weekly buckets (anchored to Sunday) by `applied_at`;
  the last bar is the current week.
- **Where they come from** = group non-wishlist apps by `source`.
- **Insight callout** priority: (1) referral conversion if ≥2 referral apps and
  ≥1 progressed; (2) top source; (3) fallback pipeline nudge.
